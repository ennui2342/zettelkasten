# The Agent System Trilemma as the Economic Logic Governing Co-Design of Architecture and Memory

The co-design principle linking multi-agent architecture to procedural memory—where role structure determines memory granularity and memory richness determines the effective ceiling of role performance—acquires its practical urgency from a deeper constraint: the Agent System Trilemma, the inherent tension among performance, cost, and efficiency in agentic AI systems. Without this trilemma, co-design would be a matter of engineering elegance. Under the trilemma, co-design becomes the primary mechanism for navigating an otherwise intractable three-way trade-off.

The connection works as follows. The trilemma establishes that maximizing task success typically demands expensive frontier models and complex multi-agent topologies, but these choices inflate both monetary cost and wall-clock latency to levels incompatible with real-world deployment. Procedural memory, accumulated and organized along the natural role boundaries of an orchestrator–worker architecture, provides a way to *shift the trilemma's frontier* rather than merely choose a point on it. When smaller or cheaper language models are augmented with well-structured procedural memory, they can recover much of the performance lost by stepping down from frontier models—while simultaneously reducing cost and, through fewer execution steps and lower per-step failure rates, improving efficiency. This is not a marginal optimization; it is a qualitative change in the feasible region of the trilemma. Memory effectively converts prior computational expenditure (past task executions) into a reusable asset that subsidizes future performance, amortizing the cost dimension across tasks over time.

Crucially, the *structure* of memory determines how much of this amortization is actually realized, and that structure is dictated by architectural choices. Orchestrator-level memory compresses strategic planning knowledge—how to decompose goals, sequence subtasks, recover from failures—into a reusable prior that reduces the number of expensive re-planning cycles and prevents the orchestrator from exploring dead-end decompositions. Task-agent memory compresses tactical execution knowledge—tool invocation patterns, API idiosyncrasies, local error handling—into guidance that reduces per-step failure rates and eliminates redundant tool calls. Both forms of compression directly attack different facets of the trilemma: orchestrator memory primarily improves efficiency (fewer planning iterations, shorter execution traces) and performance (better decomposition quality), while task-agent memory primarily improves cost (fewer wasted API calls) and efficiency (fewer retries). A flat, undifferentiated memory store would conflate these two types of compression, degrading the precision of both and weakening the trilemma-shifting effect.

The phenomenon of architectural overfitting further reinforces this logic. Deploying maximally complex multi-agent frameworks on tasks that do not require such sophistication degrades performance and inflates cost and latency—moving the system to a strictly worse point in the trilemma space. The implication is that the optimal architecture is not the most powerful one available but the one whose complexity matches the intrinsic difficulty of the task. Procedural memory plays a role here as well: accumulated experience about which decomposition strategies succeeded for which classes of tasks can inform not just *how* to decompose but *whether* to decompose at all, enabling the system to adaptively select its own level of architectural complexity. This is a form of meta-level procedural knowledge—memory about when to use the architecture's full capacity versus when to bypass it.

Retrieval strategy design interacts with the trilemma in a similarly non-trivial way. Dynamic retrieval, which performs just-in-time subtask-level memory lookups during execution, offers higher retrieval precision but incurs repeated embedding and retrieval operations that increase latency and cost. Static retrieval is cheaper and faster but less precise, particularly when orchestrator-level memory is weak. Query rewriting occupies an intermediate position, shifting fine-grained retrieval to the planning stage to avoid runtime overhead. The choice among these strategies is itself a trilemma navigation decision: the right retrieval approach depends on which dimension—performance, cost, or efficiency—is most constrained in a given deployment context, and on whether strong orchestrator memory is available to compensate for coarser retrieval.

The unifying insight is that **the co-design of architecture and memory is not merely a software engineering best practice but the principal lever for expanding the feasible region of the Agent System Trilemma**. Systems that treat architecture, memory, and retrieval as independent modules will be confined to choosing among unsatisfying trade-off points. Systems that co-design these components—partitioning memory along role boundaries, selecting retrieval strategies that match the available memory topology, and using accumulated experience to adaptively calibrate architectural complexity—can achieve combinations of performance, cost, and efficiency that would otherwise appear to violate the trilemma's constraints.
