The integration step is cascading: when a newly-split note is still broad enough that the next draft also doesn't fit cleanly, it splits again.

## Integration Reasoning (step1 and step1.5)

The `reasoning=` field in `integrate.step1` and `integrate.step1_5` logs reveals the model's decision logic in detail.

### Step 1.5 SPLIT logic

Step 1.5 fires after step 1 proposes an operation that would land a draft into a note the model believes is already too topically broad. The reasoning is consistently precise about *why* the split is warranted:

> *"The existing note focuses on procedural memory mechanisms for workflow automation... The draft introduces GraphRAG as a distinct paradigm for knowledge graph retrieval... These are two separate applications of RAG with different architectures, use cases, and challenges."*

> *"The existing note (z20260318-020) focuses on memory management as a design trade-off within enterprise agentic architectures... The draft introduces a fundamentally different framing: cognitive architectures with three distinct memory types (working, episodic, semantic)... These are separate conceptual frameworks—one is architectural optimization for LLM agents, the other is a cognitive science-inspired memory taxonomy."*

> *"The existing note (z20260318-027) focuses on benchmarks and metrics for evaluating LLM-based agent systems... The draft introduces a distinct CLASSic framework (Cost, Latency, Accuracy, Security, Stability) that reframes evaluation around deployment constraints and operational reliability... a fundamentally different lens."*

The step1.5 reasoning distinguishes between conceptual layers (foundations vs. specialized mechanisms), competing framings of the same domain (benchmark methodology vs. operational evaluation criteria), and orthogonal concerns that happen to share a parent topic. The confidence gradient in SPLITs (0.95, 0.85, 0.78, 0.72) correlates with how clearly distinct the topics are: at 0.95 the note already identified the mismatch cleanly; at 0.72 the split is more judgement-call.

The single EDIT reasoning (confidence 0.92) is different in character:

> *"The note covers a single coherent topic—evaluation methodologies for LLM-based agent systems—but has accumulated substantial detail across multiple updates. The draft introduces concrete benchmark examples and validation protocols that should be integrated by compression rather than separation."*

This is the step-1.5 path recognising that the note is large but topically unified, so compression is appropriate rather than splitting.

### SYNTHESISE reasoning: non-obvious bridging

The most interesting reasoning appears in SYNTHESISE decisions that identify connections the individual notes cannot see:

**Memory as both solution and attack surface** (paper 3):
> *"the memory abundance that resolves the agent trilemma simultaneously expands the attack surface for memory poisoning... memory-augmented agentic systems must balance performance gains against poisoning risk."*

This is a genuine logical inversion — the same property (many stored memories) is simultaneously the mechanism that solves the trilemma and the mechanism that makes poisoning more damaging. Neither the trilemma note nor the memory poisoning note states this; it emerges from holding both.

**Phase transition cascade** (paper 16, five consecutive SYNTHESISEs): Each synthesis built on the prior, creating an argument chain:
1. Scaling laws quantitatively bridge phase transitions and resource allocation
2. Information-theoretic constraints (finite context, lossy communication, shared failures) are the *cause* of phase transitions
3. Ensemble mathematics (majority maps, correlated aggregation, λ per-layer gain) is the unifying language across all three
4. Canonical topologies (star, chain, tree) are constraint-mediated structural solutions, not design choices

The fifth synthesis, spanning 7 prior notes, closes the chain:
> *"topology choice, orchestration formalism, and agent role design are co-determining factors... the topology-capability interaction being multiplicative precisely because topology determines which constraint is binding."*

This is genuine theory-building from the integration step, not retrieval.

**Trust as unifying vulnerability** (paper 3):
> *"these threat vectors converge on the trust-calibration dilemma... architectural directions for memory defense (provenance tracking, decoupled verification, anomaly detection) generalize across the full threat model."*

**Context window as unified optimization surface** (paper 20, final synthesis):
> *"all five dimensions—planning structures, memory tiers, context optimization, tool output processing, and instruction coherence—are interdependent facets of a single context policy problem."*

### UPDATE reasoning: integration vs. duplication

The UPDATE reasoning shows a consistent pattern of identifying where a draft *instantiates* existing abstractions rather than duplicating them:

> *"The draft provides the concrete technical implementation and design principles for procedural memory systems... directly addressing the memory gap identified in the existing note... the draft's detailed memory mechanisms instantiate the abstract requirement for 'memory mechanisms tailored to multi-agent coordination' mentioned in the existing note."* (confidence 0.92)

> *"The draft introduces UHAT as a formal computational model... This is a distinct theoretical contribution that complements but does not overlap with existing notes on reasoning augmentation... establishes a formal foundation for understanding what functions Transformers can compute, which is orthogonal to the empirical and architectural concerns."* → routed to CREATE instead

The distinction between UPDATE (instantiate/extend an existing note) and CREATE (orthogonal foundation) is being correctly applied. Lower confidence UPDATEs (0.78) show more negotiation:

> *"The draft note provides a higher-level architectural taxonomy and benchmark overview that complements but does not duplicate the existing note... Integrating the draft into z20260318-001 will ground those solutions within a more complete architectural framework."* (0.78)

Here the fit is looser — the reasoning justifies the UPDATE but acknowledges it's a broadening rather than a deepening.

## Notable Observations

### SPLIT is new behaviour

The previous run (same 20 papers, same model) produced 0 SPLITs across 237 operations and ended with 34 notes. This run produced 23 SPLITs and 62 notes. The SPLIT behaviour was not present in the prior run — it is a pipeline change. With 23 SPLITs and 19 CREATEs and 20 SYNTHESISEs, this run created 62 notes versus 34, confirming that the SPLIT mechanism is genuinely adding notes rather than just reorganizing.

### Much lower total operation count (87 vs 237)

The previous run's high UPDATE count (135) reflects a corpus where many papers were closely related (LLM memory systems, multi-agent architectures) and converged on the same notes repeatedly. This corpus covers more distinct areas — graph databases, scientific reasoning, logistics, formal expressivity, context engineering — so new content more often forces structural change (SPLIT or CREATE) rather than incremental enrichment (UPDATE).

### Paper 16 SYNTHESISE cascade

The Phase Transition paper (2601.17311) generated 5 consecutive SYNTHESISE operations with no CREATE or UPDATE. The paper is mathematically dense and the integration step routed all five drafts as bridge notes rather than landing them in or creating atomic concept notes. This could reflect genuine synthesis (the paper's contribution is relational, not definitional) or it could indicate Form is producing drafts at an intermediate abstraction level that triggers SYNTHESISE by default. Worth spot-checking the resulting notes.

### Zero STUB, zero NOTHING

Same pattern as the previous run. The pipeline retains a strong inclusion bias. Every draft was integrated in some form.

### One EDIT

A single EDIT on 'Benchmarking and Evaluation of LLM-Based Agent Systems' (paper 12, 2601.12307). The note had received multiple cross-paper UPDATEs and SPLITs from prior papers; this EDIT was compression — the step-1.5 trigger firing to prune the accumulated content before further updates would have bloated it.
