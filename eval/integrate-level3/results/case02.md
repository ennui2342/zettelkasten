## Multi-Agent Architectures for Workflow Automation

Modern LLM-powered agentic AI systems operate through the coordinated action of multiple heterogeneous models combined with tool invocation workflows and memory modules. These systems fall along a spectrum of automation: **handcrafted systems** have their entire configuration—LLM backbone, prompting strategies, communication protocols—manually specified, as seen in frameworks like AutoGen, AutoGPT, and ChatDev; **partially automated systems** automate specific components such as agent role assignment, prompt design, or inter-agent communication topologies; and **fully automated systems** aim to autonomously design and evolve all modules.

Architecturally, agentic systems range from single-agent designs (e.g., ReAct, which iteratively alternates between observation, action, and reasoning) to dual-agent frameworks (e.g., a manager paired with a tool agent) to hierarchical multi-agent structures with specialized sub-agents for planning, web browsing, file handling, and code execution. The dominant pattern for workflow automation uses a central orchestrator agent that decomposes a high-level goal into subtasks and delegates them to specialized task agents, each interacting with specific applications through tool-use interfaces. Frameworks such as Magentic-One and AutoGen exemplify this modular, role-based team structure.

A common limitation is that the mapping from agent roles to LLM backbones is **static and human-predefined**—a web-browser agent might be hard-coded to use GPT-4o while a planner uses a different model. This rigidity lacks adaptability to the varying difficulty and nature of sub-tasks encountered during execution. System state is typically maintained through shared memory or scratchpad mechanisms, and the action space encompasses both natural language generation and tool invocations (code interpreters, web search APIs, etc.), with a scheduler determining which agent is active at each time step.

During execution, the orchestrator generates an initial plan and iteratively selects the next subtask based on current state and observations returned by task agents. Task agents carry out assigned work by issuing commands to external environments—typically APIs in sandboxed containers—and return execution summaries. When progress stalls due to repeated states or looping, the orchestrator can trigger re-planning. Task success is determined by whether the final environment state matches the intended goal.

### Procedural Memory for Experience Reuse

A key weakness of current systems is their stateless, transactional nature: every task is solved from scratch with no reuse of prior experience. Agents may repeat the same mistakes across sessions and cannot improve over time. Closing this gap requires memory mechanisms, but not all memory types serve this purpose equally. Memory in LLM-based agent systems spans several categories: **episodic and semantic memory**, which replay information from dialogue histories and conversational interactions, and **procedural memory**, which captures reusable knowledge about how to perform tasks. Systems like A-MEM and Mem0 manage memory from ongoing conversations through interconnected note structures or evolving memory stores, but they are not designed for agentic learning or workflow automation. Another line of work targets short-term context optimization, compressing or contextualizing information within a single execution episode. What workflow automation specifically demands is procedural memory—records of how tasks were previously solved—that enables incremental skill acquisition across sessions.

Early approaches to procedural memory operate in single-agent settings. Synapse retrieves successful past full trajectories as exemplars, providing raw action sequences to guide future behavior. Agent Workflow Memory (AWM) induces frequently used subtask sequences from trajectories and consolidates them into reusable skills. Neither approach addresses the coordination challenges inherent in multi-agent systems, such as how to distribute memory across agents with different roles or how to align orchestration-level planning memory with execution-level task memory.

A modular approach suited to multi-agent settings decomposes past task trajectories into complementary memory types: **full-task memories** that capture orchestration plans and summarized execution traces, and **subtask memories** that encapsulate agent-specific behavior, tool interactions, and observations. These memory units are constructed offline from successful execution logs, using an LLM to transform raw logs into structured representations. The resulting memory bank is implemented as a vector database indexed by dense semantic embeddings (e.g., using models like text-embedding-3-large with libraries like FAISS), enabling retrieval of relevant memories at inference time through retrieval-augmented generation adapted to the multi-agent context.

### Role-Aware Memory Retrieval

During inference, memory is allocated in a role-aware fashion. Orchestrators receive full-task memories to support task decomposition, agent selection, and error recovery, while task agents receive subtask memories providing localized execution guidance. Multiple retrieval strategies adapt to this multi-agent structure with different trade-offs. **Static retrieval** uses the overall task description to retrieve full-task memories once at the start, then statically extracts subtask memories for agents. **Dynamic retrieval** performs per-step queries as new subtasks are generated, computing embeddings for each subtask and querying agent-specific memory banks in a just-in-time fashion. **Query rewriting** shifts fine-grained retrieval to the planning stage, using an LLM to anticipate subtasks and generate rewritten descriptions that pre-retrieve relevant subtask memories before execution begins, combining the precision of fine-grained retrieval with the efficiency of upfront computation.

Ablation studies reveal important principles about memory design in multi-agent teams. Orchestrator memory is critical for effective high-level planning, task decomposition, and delegation—removing it causes significant performance drops even when task agents retain their memories. Task-agent memory complements orchestrator memory by improving execution-level precision and tool-use accuracy. Fine-grained subtask retrieval strategies (dynamic and query rewriting) offer particular advantages when orchestrator-level guidance is absent or when task agents are smaller models with weaker capabilities. In full-memory settings, however, all three retrieval strategies achieve comparable overall performance, suggesting that the choice among them matters most under constrained conditions. Memory-augmented agents also exhibit reduced execution steps and lower per-step failure rates compared to stateless agents, indicating improvements in both efficiency and reliability. Notably, augmenting memories with additional reasoning traces yields only marginal differences, suggesting that well-structured modular memory already provides sufficient procedural guidance without needing explicit chain-of-thought annotations.

### Evaluation and Cross-Configuration Results

Evaluation spans a growing set of benchmarks. OfficeBench evaluates multi-step office automation tasks at varying complexity levels: single-application, cross-application, and multi-application workflows, with success assessed programmatically by comparing final environment states against expected outputs. GAIA is a general-purpose agentic benchmark with three difficulty tiers assessing file reading, web browsing, and coding capabilities. BrowseComp+ focuses on web browsing evaluation, DS-1000 targets data science tasks, HotpotQA tests multi-hop question answering via web search, and DDXPlus evaluates medical diagnostic reasoning.

Experiments across different team configurations—homogeneous teams of large models, hybrid teams pairing a large orchestrator with smaller task agents, and teams composed entirely of small language models—show that memory-augmented systems consistently outperform stateless baselines. Smaller models equipped with procedural memory can substantially narrow the gap with larger models, and hybrid teams with memory can even surpass full large-model teams that lack it. This suggests that experience reuse through well-structured, role-aware procedural memory may be as important as raw model capability for sustained performance in workflow automation. It also highlights a tension with the prevailing static assignment of models to roles: if procedural memory can compensate for weaker model capabilities in certain roles, the rigid human-predefined mapping of agents to LLM backbones may be leaving significant performance on the table, motivating research into adaptive or fully automated system design.

## Hierarchical Multi-Agent Systems

Hierarchical multi-agent systems (HMAS) have emerged as a dominant architectural pattern for tackling problems beyond the scope of individual agents. By organizing agents into structured hierarchies, these systems enable sophisticated task decomposition, parallel execution, and efficient coordination—mirroring the division of labor in effective human organizations.

Several architectural paradigms illustrate the power of this approach. The **Manager-Expert** paradigm involves a manager agent that decomposes high-level tasks and assigns sub-tasks to specialized expert agents. A key feature is feedback-driven replanning: experts report on failures and learned insights, allowing the manager to adapt strategy dynamically rather than committing rigidly to an initial plan. This closed-loop interaction between hierarchical levels distinguishes effective HMAS from simple fan-out delegation. Centralized control models, exemplified by frameworks like MetaGPT and AutoAct, use a central controller to manage predefined workflows and assign roles to specialized agents, ensuring structured and predictable processes at the cost of flexibility.

More advanced frameworks explore **automated and self-evolving hierarchies**. Pyramid-like directed acyclic graph (DAG)-based architectures with generalized "agent-as-a-tool" mechanisms allow higher-level agents to invoke lower-level agents as callable tools, enabling automatic task decomposition without requiring the hierarchy to be fully specified in advance. This recursive composition—where any agent can serve as both a principal (delegating work downward) and a tool (being invoked from above)—creates the potential for agent hierarchies to evolve and adapt over time as new specialist agents are introduced or existing ones are recombined. Workflow orchestration in such systems can be modeled as directed conditional graphs, where nodes represent agents or processing modules and edges represent possible control flow transitions, with branching logic determined at runtime based on intermediate results.

The choice of hierarchy depth and branching structure involves important trade-offs. Shallow hierarchies with a single orchestrator and many peer task agents minimize communication overhead but concentrate planning burden on one agent, creating a bottleneck when tasks are highly complex. Deeper hierarchies distribute planning across multiple levels but introduce coordination costs and potential information loss as summaries propagate upward. The optimal structure depends on task characteristics: highly decomposable tasks with independent subtasks favor wide, shallow hierarchies, while tasks with deep sequential dependencies or requiring iterative refinement benefit from deeper structures with tighter feedback loops between levels.

A recurring design tension in HMAS is the degree of autonomy granted to agents at each level. Fully centralized designs where the top-level agent micromanages every decision are brittle and scale poorly, while fully decentralized designs where agents act independently risk incoherent global behavior. Effective hierarchical systems typically strike a balance: higher-level agents set goals and constraints while lower-level agents retain autonomy over execution details, with structured communication channels (status reports, exception escalation, resource requests) maintaining alignment without requiring constant supervision.
