# Evaluation and Testing of AI Agent Safety and Behavioral Reliability

Evaluating LLM-based agents—particularly in multi-agent settings—requires moving beyond task-completion metrics. Non-determinism is a fundamental characteristic of systems built on large language models, introducing behavioral uncertainty during execution that makes evaluation especially challenging. Agents may appear to perform well while violating policies, skipping verification checks, or making uninformed decisions—and they may do so inconsistently across executions, completing a task correctly once while following entirely different paths, skipping steps, or violating constraints the next time. A comprehensive evaluation strategy must therefore combine adversarial safety benchmarks, metrics that localize vulnerabilities within complex pipelines, multi-dimensional behavioral assessment across the full agent architecture, and explicit strategies for accounting for non-deterministic behavior.

### Non-Determinism as a Structural Challenge

Non-determinism is not a marginal concern but a pervasive property that affects every evaluation dimension. Agentic systems exhibit variability in execution paths, tool selection, and memory retrieval patterns, and this uncertainty propagates through the system, affecting the correctness of tool invocations, the consistency of memory operations, and the reliability of multi-agent interactions.

Experimental evidence demonstrates this concretely: running the same scenario three times with identical configurations produces variable results in input token consumption (ranging from 7,784 to 19,335 tokens) and response times (ranging from 72 to 454 seconds), indicating scenario-dependent resource utilization patterns driven by non-deterministic model behavior. Evaluation methods that rely on binary task-completion metrics from single executions fundamentally cannot capture this variability. An agent that passes a test once may fail it the next time—not because anything changed in the configuration, but because the underlying model made different choices.

Addressing non-determinism requires multiple complementary strategies. Temperature settings control randomness in model outputs but cannot eliminate behavioral variation—uniform temperature settings may not capture the full range of behavioral divergence. Multiple independent executions with averaged results help account for variability. Static analysis provides deterministic baselines against which runtime deviations become visible. Dynamic execution monitoring captures the actual distribution of behaviors. Judge-based evaluation protocols offer qualitative assessment of reasoning quality and alignment that deterministic metrics cannot provide. Principles from self-adaptive systems research—continuous monitoring and adaptive responses to changing conditions—offer additional relevant strategies for handling environmental uncertainty.

### Multi-Pillar Behavioral Assessment

Assessment decomposes across four pillars, each addressing distinct failure sources and each subject to non-deterministic variation:

- **LLM**: Reasoning, instruction following, and safety alignment—especially for complex instructions involving multiple constraints, tools, or conditional logic. The model may not consistently adhere to predefined rules or may generate unsafe outputs across executions.
- **Memory**: Storage consistency and retrieval accuracy, where information may be duplicated, outdated, or incompletely recalled. Non-determinism manifests as variable retrieval completeness across runs.
- **Tools**: Tool selection, parameter mapping, and execution sequencing, where the agent may invoke incorrect tools or pass semantically wrong parameters in some executions but not others.
- **Environment**: Workflow definitions, guardrails, and configuration for reproducible evaluation. The environment itself is a source of operational uncertainty that influences agent behavior—environmental variability interacts with model non-determinism to widen the space of possible execution outcomes.

This decomposition matters because aggregate metrics mask severe component-level failures. In autonomous cloud operations validation, baseline metrics reported 100% task completion for a security incident response scenario, but pillar-specific metrics revealed only 13.1% memory recall and 33% tool sequence correctness. Another scenario achieved perfect tool sequencing but only 33% policy adherence—actions proceeded without consulting safety guidelines, entirely invisible to conventional metrics. When non-determinism is factored in, these component-level failures may appear or disappear across runs, making single-execution evaluation fundamentally unreliable.

### Three Evaluation Layers

**Static analysis** validates behaviors against ground-truth specifications using deterministic metrics (precision, recall, F1, BLEU-1 for memory; classification and parameter accuracy for tools). Static analysis provides the deterministic baseline essential for distinguishing genuine failures from non-deterministic variation.

**Dynamic execution** monitors runtime behaviors—tool invocation sequences, memory queries, state changes—to detect deviations that only manifest in execution context, such as skipping diagnostics before remediation or violating production-state guardrails. Because non-determinism means runtime behavior varies across executions, dynamic monitoring must span multiple runs to characterize the distribution of behaviors rather than a single trajectory.

**Judge-based evaluation** qualitatively assesses reasoning, safety, and alignment:
- *LLM-as-Judge*: Evaluates execution logs via structured prompts, producing scored assessments (~$0.06 and 15 seconds across scenarios)—suitable for continuous monitoring across multiple executions.
- *Agent-as-Judge*: An auditor agent dynamically designs capability tests, issues instructions, and verifies execution through environment inspection. Roughly 16× costlier and 62× slower than LLM-as-Judge, best reserved for pre-deployment audits where thoroughness justifies the cost.

Judge-based evaluation is particularly valuable for capturing aspects of non-deterministic behavior that deterministic metrics miss—qualitative shifts in reasoning quality, alignment drift, and inconsistent policy consultation that may not register as discrete metric changes.

### Safety Benchmarks

Adversarial benchmarks target distinct threat surfaces: **RedCode** probes harmful code generation and execution at the system level; **SafeArena** tests agent behavior on harmful tasks via offline replicas of real-world platforms (Reddit, GitLab, e-commerce); **AgentHarm** evaluates robustness against jailbreak-style adversarial prompting; **Agent Security Bench (ASB)** targets prompt injection attacks embedding malicious instructions in benign input.

### Safety Metrics and Failure Attribution

**ASR** and **Refusal Rate** provide aggregate outcomes but cannot localize responsibility within agentic architectures. **ARIA** categorizes outcomes into four risk levels (immediate refusal through full harmful completion), offering a finer behavioral spectrum without component attribution. **Dharma** localizes rejections and failures to specific pipeline components—classifying trajectories into planner-level behaviors (Planner-Stop, Planner-Ignored, Planner-Failed) and sub-agent-level behaviors, distinguishing cases where a planner warned but sub-agents ignored the warning from cases where sub-agents refused but the orchestrator overrode. Dharma localizes *safety* failures across architectural roles; pillar-specific metrics localize *behavioral reliability* failures across functional dimensions. Together they provide both safety and operational coverage.

### Failure Patterns

- **Tool orchestration** exhibits the highest failure rates in complex scenarios, primarily from skipping diagnostic or verification steps—even when task completion appears successful. Non-determinism means these failures may appear intermittently, making them especially dangerous in production.
- **Memory failures** increase with complexity and illustrate how uncertainty manifests in practice. Multi-hop and temporal reasoning queries achieve perfect precision (100%) but substantially lower recall (~27–30%): agents retrieve accurate information when they do retrieve, but non-deterministically fail to retrieve all relevant memories. This precision-recall gap creates situations where agents make decisions based on incomplete context despite having access to the necessary information—a failure mode invisible to precision-only metrics.
- **Environment guardrail violations** appear specifically in multi-agent scenarios where production states change despite protection constraints.

### CloudOps as Evaluation Domain

Autonomous cloud operations provides a demanding evaluation domain where agents must consult policies before acting, perform diagnostics, retrieve context, invoke tools correctly, and respect guardrails—with incorrect actions risking security breaches, cost overruns, or service disruptions. Representative scenarios include **cost optimization** (terminating underutilized resources while respecting non-production prioritization and CAB approval), **security incident response** (remediating publicly accessible storage with compliance checking and logging before policy changes), and **multi-agent root cause analysis** (coordinating specialized agents for temporal correlation rather than symptomatic scaling). Environment configuration requires domain-expert-designed workflows, resettable known states, fault injection, and guardrails enforced through IAM policies and role-based access control. Observability spans both platform-level telemetry (e.g., AWS CloudTrail) and agent-level capture of tool calls, memory operations, and policy checks—enabling prompt refinements and root cause analysis when agents fail.

### Evaluation Infrastructure

Systematic safety evaluation faces integration complexity, insufficient multi-agent metrics, and lack of reproducible comparison infrastructure. The SafeAgents framework addresses these through modular, framework-agnostic abstractions (Agent, Tool, Team classes) exposing design decisions as configurable parameters, with a unified factory method enabling consistent experimentation across frameworks and benchmarks. Reproducibility is especially critical given non-determinism: evaluation infrastructure must support multiple independent executions with controlled configurations and aggregated results to produce statistically meaningful assessments. Automated test case generation from agent capabilities remains a promising research direction.

### Architectural Implications for Safety

Hallucinations occur almost exclusively under ReAct-style prompting in multi-agent configurations (e.g., Claude Sonnet 4: 36% hallucination rate with multi-agent ReAct, 0% otherwise); function calling produces more constrained, verifiable behavior. This coarse architectural choice dominates fine-grained safety tuning, reinforcing that high-level design decisions shape the failure landscape all other evaluation dimensions must cover. These architectural choices also interact with non-determinism: less constrained architectures amplify the range of non-deterministic behaviors, while more structured approaches (function calling, explicit tool schemas) narrow the distribution of possible execution paths. Production deployment experience confirms that agents may complete tasks while exhibiting behavioral deviations undetected by standard methods—surfacing only at runtime, not in static validation—highlighting the gap between prototype performance and production readiness.
