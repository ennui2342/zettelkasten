# Context Engineering as State Compression for Large Language Model Agents

Large Language Model (LLM) agents operating in complex, multi-step workflows face a persistent bottleneck: managing the evolving context that accumulates during execution. An LLM agent's state is represented not by model parameters but by its prompt context—system instructions, evolving plans, previous reasoning traces, tool results, user instructions, long-term memories, retrieved knowledge, and intermediate outputs. As tasks grow in depth and breadth, this state becomes increasingly large, noisy, redundant, and expensive to process.

Even models with very large context windows (200k–1M+ tokens) exhibit degraded reasoning quality when overloaded with irrelevant or poorly structured information—a phenomenon sometimes called "context rot." Modern agentic failures are overwhelmingly context failures rather than model failures. Agents fail when crucial information is dropped or buried in irrelevant text, when irrelevant details overload the model's attention, when multiple tasks compete for context bandwidth, when instructions contradict or drift over time, or when context becomes too long to process efficiently.

**Context engineering** is the discipline of continuously optimizing what an agent sees at each step of execution. This goes beyond prompt engineering (which optimizes initial instructions) and retrieval-augmented generation (which optimizes what is retrieved). Context engineering involves learning a state compression policy over an agent's evolving execution state, treating context as a latent execution state that must be curated, transformed, and compressed to maintain fidelity, avoid attention dilution, and reduce inference cost.

### Plan-Aware Compression

A plan-aware approach to context engineering conditions compression on multiple future plan steps and the global workflow structure, rather than optimizing for a single upcoming query. This involves several interrelated capabilities:

- **Context pruning** — removing irrelevant content
- **Summarization** — condensing verbose information
- **Rewriting** — restructuring content for clarity
- **Compression** — reducing token count while preserving meaning
- **Re-injection** — restoring previously compressed information when needed
- **Task refinement** — updating instructions to remain consistent with compressed context

Plan-aware relevance can be operationalized through explicit signals evaluated at the trajectory level, including semantic equivalence between full- and compressed-context outcomes, per-step compression ratios, implicit coverage of upcoming plan steps, and preference judgments from evaluators.

### Compression as Regularization

A key finding is that compression can act as a form of regularization. Removing irrelevant or contradictory information forces agents to reason over more coherent, structured states, reducing distraction and mitigating the risk of attending to spurious details. This means plan-aware compression can actually improve task correctness beyond what uncompressed full-context execution achieves—a counterintuitive result that reframes compression not merely as a cost-saving measure but as a reasoning enhancement.

Additionally, conditioning compression on upcoming tasks helps mitigate **instruction drift**, a common failure mode in long-running workflows where accumulated context causes the agent's behavior to diverge from its original plan. By continuously reshaping context to foreground plan-relevant information, the agent maintains alignment with its intended trajectory even across many execution steps.

### Cumulative Attention Cost

Cumulative attention dependency—measured as the sum of input tokens across all steps—serves as a practical metric for total transformer attention cost, correlating with latency and the quadratic compute growth inherent to self-attention. Reducing this metric through intelligent compression directly translates to lower inference costs and faster execution. In multi-step workflows where context is re-processed at every turn, even modest per-step compression compounds into substantial savings across the full execution trace.

### Relationship to Memory and Architecture

Context engineering intersects deeply with agent memory design. Working memory—the transient task state held in the prompt—is precisely what context engineering operates on. The distinction between context engineering and memory management is one of temporal scope: memory systems decide what to retain across tasks, while context engineering decides what to surface within a task at each step. In practice, the two are tightly coupled. Effective context compression depends on the ability to offload information to external memory stores and re-inject it selectively, while effective memory retrieval depends on understanding what the current compressed context lacks.

In multi-agent architectures, context engineering becomes even more critical. Each agent in an orchestrator–worker topology maintains its own context window, and the orchestrator must decide not only how to decompose tasks but how to package context for each worker. Poorly compressed orchestrator-to-worker handoffs waste worker context capacity on irrelevant planning history, while overly aggressive compression risks stripping information the worker needs. The communication budget between agents—formalized in multi-agent reasoning frameworks as inter-node message events in a DAG—is fundamentally a context engineering problem: each message must compress the sender's state into a form maximally useful for the receiver's upcoming computation.
