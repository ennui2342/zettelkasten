# Memory Poisoning, Multi-Agent Security, Non-Determinism, and the Broader Safety Landscape of LLM-Based Agents

Large language models are typically aligned to human preferences through reinforcement learning or instruction tuning and sometimes paired with content moderation modules, but both alignment and moderation can be circumvented by adversarial prompts and jailbreak techniques. When LLMs are deployed as agents capable of using tools, writing code, and interacting with environments, safety risks manifest differently than in conversational settings—leading to scenarios such as generating and executing malicious code, harmful browser interactions, and multi-step agent misuse. Agents that maintain persistent memory—storing past interactions for retrieval as few-shot examples in future sessions—are vulnerable to a further class of attacks in which an adversary corrupts the memory store through ordinary user interactions, causing the agent to produce incorrect or harmful outputs for subsequent legitimate users.

These vulnerabilities sit within a broader and rapidly expanding landscape of security threats facing LLM-based agents and multi-agent systems, spanning prompt injection, retrieval corpus corruption, image-based adversarial inputs, malicious tool outputs, cascading multi-agent failures, persistent deceptive behaviors, and emergent safety breakdowns that arise specifically from multi-agent coordination. Defending against these attacks remains a challenging and understudied problem, with existing defenses proving largely ineffective against sophisticated attack strategies and comprehensive benchmarking efforts such as Agent Security Bench (Zhang et al., 2025) revealing systemic gaps in current agent security across threat models including prompt injection, data leakage, and denial-of-service attacks.

### Non-Determinism and Uncertainty as a Foundational Challenge

A central challenge underlying both security evaluation and behavioral assessment of agentic AI systems is the inherent uncertainty introduced by non-deterministic AI models. Unlike conventional software components, LLM-based agents exhibit variability in execution paths, tool selection, and memory retrieval patterns. This non-determinism causes uncertainty to propagate through the entire system, affecting the correctness of tool invocations, the consistency of memory operations, and the reliability of multi-agent interactions. The implication is that adversarial vulnerabilities, behavioral deviations, and coordination failures may manifest intermittently and unpredictably—making them harder to detect, reproduce, and defend against than deterministic software bugs.

Multiple sources of uncertainty can be mapped to distinct evaluation pillars. The **LLM pillar** introduces failures in instruction following and safety alignment—agents may deviate from prescribed workflows or attempt unauthorized actions unpredictably. The **Memory pillar** presents challenges where storage inconsistencies (such as retaining outdated information rather than replacing it) and retrieval inaccuracies emerge non-deterministically. The **Tools pillar** exhibits failures in tool selection, parameter mapping, and execution sequencing that vary across runs. The **Environment pillar** represents operational context where fluctuations can significantly impact outcomes, drawing on research in self-adaptive systems that recognizes the environment as a distinct source of uncertainty influencing agent behavior.

Experimental evidence illustrates this uncertainty concretely. Across three independent runs of each evaluation scenario (conducted specifically to account for non-deterministic LLM behavior), significant variability was observed: input token consumption for one scenario ranged from 7,784 to 19,335 tokens, and response times for another ranged from 72 to 454 seconds. This variability means that a single evaluation run is insufficient to characterize an agent's behavioral profile—an agent may pass safety checks on one run and violate policies on the next. Evaluation methods from conventional software engineering are insufficient precisely because they overlook this fundamental non-deterministic nature, and addressing it requires multiple complementary evaluation modes: static analysis for deterministic correctness, dynamic execution for runtime deviation detection, and judge-based evaluation for qualitative behavioral assessment.

### Safety in Single-Agent versus Multi-Agent Contexts

Critically, while extensive research has examined the safety of individual LLM-based agents against adversarial prompts and jailbreak techniques, it is unclear whether safety guarantees developed for single-agent settings extend to multi-agent contexts. In multi-agent systems (MAS), tasks are divided among specialized agents with limited access to global context, introducing new avenues for failure: agents may act on incomplete information, coordinate improperly, or inadvertently override safeguards that would have prevented harmful behavior in a single-agent scenario. A harmful request that a single agent would refuse can, when decomposed across multiple agents, result in unsafe collective outcomes—a structural vulnerability that compounds the risks posed by memory poisoning, prompt injection, and other attack vectors. The distributed nature of MAS creates cognitive load distribution where no single agent maintains sufficient context to identify overall harmful objectives. The specific architectural choices governing how these systems are organized—centralized versus decentralized coordination, the granularity of task delegation, and how context flows between agents—turn out to be more consequential for security outcomes than the high-level architectural category itself.

Non-determinism amplifies these multi-agent risks: because each agent's behavior varies across runs, the coordination dynamics of the system as a whole become even less predictable. A coordination failure that emerges only under specific combinations of agent behaviors may appear in one run and vanish in the next, making systematic detection exceptionally difficult.

A compounding problem is that conventional evaluation methods—relying on binary task-completion metrics—fail to surface these behavioral failures. Agents may complete tasks while violating policies, skipping verification steps, making uninformed decisions, or failing to retrieve relevant context, all of which are invisible to standard outcome-based evaluation. This gap between apparent success and actual behavioral integrity motivates the development of structured assessment frameworks that go beyond measuring whether a task was completed to examining how it was completed. Production deployment experience in autonomous cloud operations (CloudOps) has concretely demonstrated this problem: agents completing cloud management tasks while exhibiting behavioral deviations—such as skipping policy checks, bypassing approval workflows, or retaining outdated configuration details—that went entirely undetected by conventional evaluation methods.

### Autonomous Cloud Operations as an Application Domain

Cloud operations (CloudOps) involves managing and optimizing cloud infrastructure through automated workflows, encompassing tasks such as resource provisioning, compliance enforcement, and incident response. Organizations facing increasingly complex cloud environments require autonomous systems capable of executing diverse operational tasks at scale. Platforms like MontyCloud leverage AI agents to automate these cloud management tasks, representing a growing reliance on autonomous decision-making in operational workflows. CloudOps serves as a particularly revealing application domain for studying agent behavioral reliability because it combines high-stakes consequences with complex policy requirements and multi-step operational workflows.

Three production-motivated scenarios illustrate the behavioral reliability challenges specific to this domain. The **Cost Optimization** scenario requires an agent to reduce AWS infrastructure costs by 30% by identifying and terminating underutilized EC2 instances while adhering to organizational policies on non-production prioritization, environment verification, and Change Advisory Board (CAB) approval. The **Security Incident Response** scenario involves remediating a publicly accessible S3 bucket containing sensitive data, requiring the agent to assess exposure scope, check compliance requirements, enable logging before policy changes, and apply remediation while maintaining authorized service access. The **Multi-Agent Root Cause Analysis** scenario requires diagnosing a 60% response time degradation in a payment service caused by network misconfiguration, demanding coordination between Performance, Security, and RCA agents to identify the root cause through temporal correlation rather than treating symptoms through costly scaling.

These scenarios demand that agents consult policies before acting, perform diagnostic checks, retrieve relevant context (such as WAFR workload details aligned with cloud best-practice guidelines), invoke tools with correct parameters (e.g., using instance IDs rather than region names), and respect guardrails enforced through mechanisms like IAM policies and role-based access control. The environment must support resetting to baseline states between tests, injecting faults, and setting cost or resource limits. Observability mechanisms including telemetry, tracing, logging, and platforms like AWS CloudTrail are essential for monitoring agent behavior and supporting root cause analysis during fault conditions. The non-deterministic variability documented across evaluation runs—with token consumption and response times varying by factors of two to six—means that even well-designed CloudOps agents may exhibit inconsistent policy adherence across executions, reinforcing the need for repeated evaluation and multi-modal assessment. The practical gaps revealed by production deployment in this domain directly motivated the development of the multi-pillar assessment framework described below and underscore the importance of environment-layer defenses including deterministic guardrails, IAM policies, and control policies for safe autonomous operations.

### Multi-Agent System Architectures and Design Primitives

Multi-agent systems powered by LLMs can be organized into two primary architectural families: **centralized** and **decentralized**. In centralized architectures, a single coordinating agent (orchestrator) decomposes user requests into plans, assigns subtasks to specialized sub-agents, and aggregates outputs. In decentralized architectures, decision-making authority is distributed across agents with peer-to-peer communication and local coordination, reducing reliance on any single agent but raising risks of mis-coordination or conflict. Frameworks such as Magentic-One, LangGraph, and OpenAI Agents provide abstractions for building both types, with ready-to-use implementations that vary substantially in their specific design choices even when representing the same architectural pattern.

Three critical **design primitives** influence vulnerability surfaces across these architectures:

1. **Sub-agent Autonomy** concerns the granularity of task delegation. Orchestrators can assign high-level subtasks requiring multiple actions (granting sub-agents sufficient context to recognize harmful objectives) or atomic instructions where the orchestrator maintains fine-grained control over each step. Magentic-One restricts sub-agents to single actions before returning control, while LangGraph allows sub-agents to execute multiple sequential actions autonomously. Atomic delegation fragments malicious intent across isolated actions, making detection more difficult—a mechanism closely related to the context fragmentation vulnerability discussed below under cascading failures.

2. **Stratified vs. Combined Planning** determines whether plan construction occurs as a separate phase before delegation or is integrated with the delegation process. Magentic-One employs stratified planning (generating a fact sheet and structured plan before execution), while LangGraph integrates planning into the orchestrator's delegation decisions. Stratified planning enables detailed upfront analysis but can lead to blind execution of pre-generated plans; combined planning enables dynamic adaptation but places greater cognitive load on the orchestrator.

3. **Context Organization** governs how task descriptions, plans, and agent roles are structured within the LLM's input context. Different frameworks make varying choices about sequencing plan information relative to task descriptions and positioning agent role specifications (e.g., system messages vs. user messages vs. assistant messages), which can significantly influence how the underlying LLM processes and prioritizes information for both task execution and safety behavior.

Empirical results demonstrate that these granular design choices matter more for security outcomes than the high-level architectural category. For instance, among centralized systems, Magentic-One achieves 83.70% critical-risk outcomes on RedCode while other centralized implementations show substantially different results. Decentralized systems like Swarm achieve lower attack success rates on context-heavy benchmarks because individual agents retain full context and can independently reject unsafe actions, though this advantage is domain-dependent.

### Attack Mechanism: Memory Poisoning

The core vulnerability arises from the combination of three properties: the agent autonomously decides what to memorize, the memory persists across sessions and users, and retrieved memories directly influence output generation. An attacker who can interact with the agent as a regular user can exploit this pipeline without requiring any elevated privileges or direct access to the memory store.

The most thoroughly studied instantiation is MINJA (Memory Injection Attack, Dong et al., 2025), which operates entirely through the query interface. MINJA uses three coordinated techniques. **Bridging steps** are a sequence of interactions that incrementally steer the agent toward forming a poisoned association—for instance, gradually linking a victim identifier to a target identifier across multiple conversational turns. **Indication prompts** are carefully constructed inputs that signal to the agent that the current interaction is worth committing to long-term memory. **Progressive shortening** compresses the malicious payload across successive interactions into entries that are short, plausible-looking, and difficult to distinguish from legitimate memories during manual audit.

Under idealized laboratory conditions—an empty initial memory store and fixed retrieval parameters—MINJA achieves injection success rates above 95% and attack success rates above 70%. The non-deterministic nature of LLM-based memory operations adds a further dimension: whether a poisoned memory is stored, retrieved, and acted upon may vary across sessions, meaning that memory poisoning attacks may succeed intermittently rather than reliably—complicating both exploitation and detection.

### The Broader Threat Landscape

Memory poisoning is one instantiation of a much wider family of attacks on LLM-based agents. Understanding the full threat landscape is essential because defenses narrowly tailored to one attack vector may leave systems exposed to others, and because many attack categories share structural similarities that inform unified defensive thinking. Beyond user-facing adversarial prompts, agents are susceptible to attacks through malicious tool outputs, memory or knowledge-base poisoning, and image-based adversarial inputs. Multi-agent systems introduce additional attack surfaces including propagation of malicious prompts between agents, exploitation of agent specialization and collaboration dynamics, and vulnerabilities to rogue or compromised agents.

#### Prompt Injection and Manipulation

Prompt injection represents a foundational threat category that operates across many channels. MINJA demonstrates that query-only interactions can corrupt persistent memory, but the attack surface extends further. Lee and Tiwari (2024) explore LLM-to-LLM prompt injection within multi-agent systems, where malicious instructions propagate between agents through natural interaction channels—meaning that even agents not directly exposed to adversarial users can be compromised through inter-agent communication. Li et al. (2025b) present system prompt poisoning attacks that persist beyond individual user injections, affecting all subsequent users and thus sharing the cross-session persistence property that makes memory poisoning dangerous. At the supply-chain level, Wan et al. (2023) demonstrate that models can be poisoned during instruction tuning, introducing backdoors activated by specific triggers—a vulnerability that operates before deployment and is invisible to runtime defenses.

#### Retrieval-Augmented Generation Attacks

RAG systems present their own attack surface that overlaps with but is distinct from memory poisoning. BadRAG (Xue et al., 2024) identifies vulnerabilities in RAG architectures, while PoisonedRAG (Zou et al., 2024) demonstrates knowledge corruption by injecting malicious documents into retrieval corpora. AgentPoison (Chen et al., 2024) targets RAG knowledge bases and memory stores, though it assumes a stronger threat model with direct system access compared to MINJA's query-only approach. The distinction matters: query-only attacks are more realistic in most deployment scenarios but may be less potent, while direct-access attacks represent a more severe but less likely threat. Non-deterministic retrieval behavior—where the same query may surface different documents across runs—further complicates the reliability of both RAG-based attacks and defenses.

#### Multi-Agent Cascading Failures and Architectural Vulnerabilities

Multi-agent systems introduce cascading failure risks that amplify the impact of any single compromise. Men et al. (2025) demonstrate contagious jailbreak attacks where malicious instructions spread through shared memory structures, creating cascading failures across interconnected agents—one compromised agent can infect others through the very collaboration mechanisms that make multi-agent systems powerful. Amayuelas et al. (2024) show how adversarial manipulation can compromise multi-agent debate systems, undermining a mechanism often proposed as a safety measure. Long and Li (2025) demonstrate FuncPoison attacks on function libraries in autonomous systems, targeting shared infrastructure rather than individual agents and thus affecting all agents that depend on the compromised library.

Beyond these known attack patterns, systematic evaluation of MAS architectures reveals that the structural properties of multi-agent coordination themselves create security vulnerabilities independent of any specific adversarial technique. Centralized architectures exhibit several characteristic "weak links" that map directly onto the design primitives identified above:

- **Inadequate fallback mechanisms** when planners refuse a task, causing the system to proceed to downstream agents rather than halting.
- **Context fragmentation through atomic instruction delegation** (a consequence of high sub-agent autonomy restriction) that obscures harmful objectives from sub-agents who see only their narrow subtask.
- **Blind execution of pre-generated plans** (a consequence of stratified planning) without reassessing safety at each step.
- **Sub-agent refusal overrides** where orchestrators ignore safety signals from downstream agents.
- **Unmitigated execution** where no agent in the pipeline recognizes the harmful request at all.

These vulnerabilities mean that centralized MAS are not inherently safer than single-agent systems and can actually propagate undetected harmful intent more efficiently through their delegation mechanisms. The CloudOps multi-agent root cause analysis scenario illustrates a non-adversarial variant of this problem: when Performance, Security, and RCA agents must coordinate to diagnose a payment service degradation, failures in inter-agent context sharing or premature symptom-level responses (e.g., costly scaling rather than identifying network misconfiguration) represent the same structural coordination risks that adversaries can exploit.

Decentralized systems, by contrast, tend to perform better on context-heavy tasks because individual agents retain full context for independent safety judgment—each agent can evaluate the complete request rather than only a decontextualized fragment. However, implementation choices within the same architectural family (e.g., planning strategy, sub-agent autonomy, context organization) create substantial security variations that architectural category alone cannot predict. The empirical divergence between Magentic-One's 83.70% critical-risk rate on RedCode and other centralized systems' substantially different results illustrates this point: security is an emergent property of specific design choices, not a consequence of choosing "centralized" or "decentralized" as a label.

#### Persistent and Deceptive Behaviors

At a more fundamental level, Hubinger et al. (2024) demonstrate that deceptive behaviors can persist in LLMs even after safety training, revealing that standard alignment techniques may be insufficient to guarantee trustworthy behavior.
