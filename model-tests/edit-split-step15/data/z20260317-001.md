# Multi-Agent LLM Systems: Architecture, Memory, and Collaborative Reasoning

Modern artificial intelligence is shifting from generative models that produce static outputs toward **agentic AI**, in which systems autonomously perceive their environment, reason about goals, plan sequences of actions, execute those actions using tools, and learn from feedback. Multi-agent LLM systems represent a key instantiation of this shift, leveraging multiple AI agent instances working collaboratively, drawing inspiration from both software engineering principles of modular decomposition and human group problem-solving dynamics. While multi-agent systems have well-established foundations in AI research, contemporary implementations augment classical coordination principles with LLM-based reasoning, memory mechanisms, and environment interaction capabilities, enabling collaborative automation of complex workflows. These systems span two complementary paradigms: **workflow automation**, where specialized agents coordinate to execute multi-step productivity tasks, and **collaborative reasoning**, where agents engage in structured debate and discussion to improve problem-solving quality. Both paradigms share fundamental architectural concerns around agent coordination, communication topology, and the mechanisms that enable effective collaboration—concerns that are increasingly understood through the lens of formal control theory and unified agent taxonomies.

### The LLM-Based Agent as a Control System

A productive framing treats the LLM-based agent as a dynamic control system operating within a Partially Observable Markov Decision Process (POMDP), formalized as a tuple of state space, observation space, internal memory space, action/tool space, and policy. At each time step the agent cycles through four linked functions:

1. **Perception** — a function that maps partial environmental observations through multimodal encoders (progressing from text-only to grounding via screenshots, video, audio, and 3D point clouds).
2. **Memory update** — a mechanism that combines new observations with prior reasoning traces and execution feedback, subsuming retrieval-augmented generation as a special case.
3. **Cognitive planning** — a step that produces a latent reasoning trace or plan via probabilistic inference parameterized by the LLM.
4. **Action policy** — selection and execution of tool calls conditioned on the plan, with environmental feedback closing the loop.

This control-loop formulation applies equally to individual agents within a multi-agent system and to the orchestrator that coordinates them, providing a unified vocabulary for reasoning about single-agent cognition and multi-agent coordination.

### A Unified Taxonomy of Agent Design

Recent work proposes decomposing agent architectures along six interrelated dimensions, which clarify the design space for both single-agent and multi-agent systems:

**Core components** include perception (multimodal grounding), memory (discussed extensively below), action interfaces (spanning constrained API calls, code-as-action where executable scripts serve as the control surface, agent–computer interfaces like SWE-agent, native computer-use via screenshots and mouse/keyboard, and embodied vision-language-action models), and profiling (system prompts or personas that define agent identity and constrain behavior—directly relevant to OptAgent's use of crafted reasoning-strategy profiles).

**Cognitive architecture** covers planning and reflection. Planning has evolved from linear ReAct loops that interleave reasoning with action, through branching search methods like Tree of Thoughts and Monte Carlo Tree Search (LATS), to hierarchical decomposition (ReAcTree, GoalAct) and inference-time reasoning models (OpenAI's o1/o3 line) that internalize search under controllable compute budgets. Reflection mechanisms—Reflexion (verbal reinforcement from stored critiques), Self-Refine (iterative generate–critic–revise), CRITIC (tool-interactive validation), and PALADIN (recovery from tool failures)—allow agents to critique and repair their own trajectories. These reflection mechanisms are the single-agent analogues of the multi-agent debate and feedback loops discussed below.

**Learning paradigms** span ephemeral in-context learning via few-shot prompting and chain-of-thought, permanent weight updates through agent-specific fine-tuning (Agent-FLAN, FireAct), alignment via RLHF and RLAIF with process-level reward models (AgentRM, AgentPRM), self-improvement pipelines that bootstrap from failures, and non-parametric skill accumulation as in Voyager's reusable code library. OptAgent's verbal reinforcement learning and LEGOMem's procedural memory accumulation can be situated within this broader learning landscape.

### Workflow Automation Through Agent Coordination

In the workflow automation paradigm, complex multi-step tasks such as document editing, email handling, and calendar scheduling are decomposed into subtasks handled by specialized AI agents coordinated by a central orchestrator. A hierarchical architecture—where an orchestrator agent plans, delegates, and re-plans based on task agent observations—allows role specialization and scalable reasoning, though current systems remain largely stateless across executions. Addressing this statelessness is a central concern of memory systems research for LLM-based agents, and recent work has converged on multi-component hybrid memory architectures that draw on cognitive science models of human memory.

More complex workflow scenarios push beyond simple task delegation into genuine multi-agent collaboration. A representative example involves three specialized agents—Performance, Security, and Root Cause Analysis (RCA)—collaboratively diagnosing a service degradation issue. Such scenarios test whether the system can identify a root cause (e.g., a network misconfiguration) through temporal correlation rather than treating symptoms through costly scaling, requiring agents to share information and coordinate their analyses across role boundaries. These scenarios surface coordination challenges that single-agent designs never encounter.

### Controllable Orchestration and Flow Engineering

A major practical theme across both paradigms is the shift from unconstrained autonomous loops toward **controllable orchestration and flow engineering**. Rather than letting agents freely decide next steps, production systems increasingly model workflows as explicit state machines or graphs where nodes represent LLM invocations or tool calls and edges represent permissible transitions. Frameworks like LangGraph implement graph traversal with state persistence, checkpoints, and controlled cycles, while OpenAI Swarm emphasizes lightweight agent handoffs and routines. This graph-based approach improves debuggability, enables insertion of guard nodes and human approval steps, and interacts directly with safety by bounding what actions are possible and where verification occurs.

This design philosophy resonates with OptAgent's graph-based formulation of multi-agent collaboration, where agents are nodes and communications are edges—but whereas OptAgent optimizes the graph dynamically via reinforcement learning, flow-engineering approaches favor explicit, human-designed topologies with controlled flexibility. The tension between these approaches—dynamic optimization versus static controllability—is a central design trade-off in production multi-agent systems.

At the tool-integration layer, the **Model Context Protocol (MCP)** standardizes how tools and resources are exposed to agents, supporting governance patterns such as allowlists, authentication, and audit logging at protocol boundaries. This standardization is particularly important in multi-agent settings where multiple agents may access overlapping tool sets with different authorization levels. Complementing MCP at the agent-to-agent layer, communication protocols such as **A2A (Agent-to-Agent)** enable capability discovery via Agent Cards, providing a foundation for both runtime coordination and systematic evaluation of multi-agent systems.

### Failure Modes in Multi-Agent Coordination

Multi-agent scenarios reveal unique failure patterns not present in single-agent settings, and understanding these patterns is essential for building robust systems. Empirical studies have identified several characteristic failure categories:

- **Tool orchestration failures** are most acute in multi-agent scenarios (averaging 7.67 failures in complex coordination tasks), primarily from agents skipping diagnostic or verification steps before executing remediation actions. The pressure to act on information received from peer agents appears to short-circuit the careful tool-use sequences that single agents more reliably follow.
- **Memory failures** increase with scenario complexity, reaching 3.67 average failures in multi-agent cases, reflecting difficulties in recalling previous role mappings and configuration changes across agents. This finding directly motivates the memory architecture research discussed below—particularly the need for shared or distributed memory that maintains coherence across agent boundaries.
- **Environment guardrail violations** appear exclusively in multi-agent scenarios (2.00 average failures), where production instance states change despite protection constraints. This is a particularly concerning finding: coordination between agents can lead to **emergent policy violations** that individual agents would not produce alone, suggesting that safety analysis must account for compositional effects rather than verifying agents in isolation.
- **Error propagation** across agents—where one agent's failure cascades to affect others—represents a systemic risk that grows with the number of coordinating agents and the depth of their interdependencies.

Research by Cemri et al. on why multi-agent LLM systems fail identifies gaps in specifications, alignment, and verification that remain underexplored in current evaluation approaches, reinforcing the need for evaluation methods specifically designed for multi-agent coordination.

### Collaborative Reasoning Through Multi-Agent Debate

In the collaborative reasoning paradigm, multiple LLM instances engage in structured interaction to solve problems that benefit from diverse perspectives and iterative refinement. Existing approaches include predefined collaboration structures (AutoGen, ChatEval), group discussion with confidence-based consensus (ReConcile), and group debate with meta-summarizers (MAD). Simple debate setups involve non-profiled agents sharing reasoning and reflecting on each other's outputs over multiple rounds, with majority voting determining the final answer.

A key limitation of these approaches is that their interaction schemas are fixed or simplistic, potentially suppressing correct but less dominant agent contributions. GPTSwarm conceptualizes multi-agent systems as computational graphs inspired by the "Society of Mind" concept and uses reinforcement learning to optimize agent interactions, but focuses purely on performance metrics while neglecting debate quality.

**OptAgent** addresses these limitations by proposing a verbal reinforcement learning framework that models multi-agent collaboration as a graph where agents are nodes and communications are edges. The framework explicitly optimizes for communication quality—including wording clarity and logical coherence—alongside correctness. It employs agent profiling with seven manually crafted profiles reflecting different reasoning strategies (e.g., deductive logic, intuition, domain expertise), and uses force decoding to generate diverse initial outputs. Two meta-agents drive optimization: a *feedback agent* that evaluates interaction quality based on correctness and reasoning coherence, and an *action agent* that modifies the collaboration graph by creating, keeping, or deleting connections. Connection scores are initialized using self-evaluated confidence scores and updated through the reinforcement learning process, with the final answer determined by majority voting across all agents.

### Benchmarking and Evaluation of Multi-Agent Systems

Assessing the effectiveness of multi-agent systems requires evaluation methodologies that span multiple task types, difficulty levels, and metrics—and that account for the coordination-specific failure modes described above. OptAgent and related systems are evaluated across several domains:

- **Mathematical reasoning:** GSM8K (grade school arithmetic), MATH (high school-level problems across six fields), and two adversarial variants—AdversarialGSM (with three difficulty levels based on number magnitude changes) and GSM-PLUS (a robustness benchmark). For datasets with category structure (MATH, GSM-PLUS), stratified sampling ensures representation across question types.
- **Science reasoning:** ARC and GPQA.
- **Creative writing:** Following the setup from Tree-of-Thoughts work.
- **Sorting tasks:** Randomly generated numerical sequences of lengths 8, 16, and 32.

Evaluation metrics are task-specific: accuracy for math and science reasoning, coherence scores (rated 1–10 by a separate LLM evaluator assessing narrative flow, consistency, and relevance) for creative writing, and error scope (sum of wrongly sorted and missing elements) for sorting.

A critical piece of the evaluation methodology is an **upper-bound analysis** using a "choose-best" strategy, which reveals that on easier datasets like GSM8K, the model can always produce the correct answer in at least one of multiple attempts, while on harder datasets like MATH, the upper bound drops dramatically. This analysis contextualizes the multi-agent framework's improvement potential and explains why performance gains vary across datasets—the gap between actual and upper-bound performance defines the room available for collaborative refinement.

Experimental results show OptAgent outperforms single-agent methods and state-of-the-art multi-agent frameworks across mathematical reasoning, creative writing, and science reasoning tasks, using GPT-3.5-turbo, GPT-4o, and LLaMA 3.1-70B as backbone models. The adversarial datasets specifically test robustness, and OptAgent demonstrates consistent margins of improvement on these challenging variants. Performance gains are most prominent in 5-agent scenarios, while scaling beyond 5 agents yields diminishing returns.

Notably, on tasks requiring direct knowledge retrieval (ARC) or complex planning (sorting), multi-agent debate frameworks including OptAgent show limited benefit. This suggests that **collaborative reasoning is most effective for tasks requiring step-by-step logical deduction rather than factual recall or procedural planning**—a finding with important implications for system design, as it indicates that multi-agent debate should be selectively deployed based on task characteristics.

Ablation studies reveal that removing interaction quality evaluation from training hurts performance (especially with more agents), forced generation with stochastic decoding is important for reasoning diversity, and reconsidering minority answers through group discussion improves results where gaps exist between actual and upper-bound performance. The framework discovers task-specific optimal interaction patterns—for instance, the Explainer agent produces the most productive interactions but in different orders depending on the task. Limitations include high computational cost from multiple model instances and API calls, dependency on backbone model capabilities, and challenges on tasks like sorting where agents struggle to generate useful explanations for discussion.

#### Agent-as-Judge Evaluation for Multi-Agent Systems

The **Agent-as-Judge** evaluation protocol is particularly relevant for multi-agent systems, leveraging agent communication protocols such as A2A that enable capability discovery via Agent Cards. In this approach, an auditor agent designs capability tests based on each worker agent's declared capabilities, issues test instructions, and verifies execution through environment inspection rather than relying solely on self-reported outputs. In multi-agent scenarios, this protocol reveals failures that single-agent testing misses—including the emergent guardrail violations and error propagation patterns described above. The protocol's thoroughness comes at a cost: in one studied multi-agent scenario, Agent-as-Judge evaluation accounted for the majority of evaluation expense ($0.61 out of $0.96 total) due to the extensive verification cycles required for multi-agent coordination. Despite this cost, the protocol's ability to surface coordination-specific failures makes it a valuable complement to traditional benchmarking approaches.

### Memory as a Cognitive Architecture

Memory is a fundamental prerequisite for intelligent agent behavior across both paradigms, enabling context maintenance, learning from experience, and avoiding repeated mistakes. Within the POMDP formulation, memory constitutes the internal state that the memory-update function maintains across time steps, bridging the gap between partial observations and effective action selection. Agent memory is typically conceptualized in two broad temporal forms: **short-term (working) memory** holding immediate task context, limited by the LLM's context window, and **long-term memory** retaining information across sessions via external data stores like vector databases. Beyond this temporal distinction, memory systems can be categorized by the *type* of knowledge they store.

**Episodic and semantic memory** approaches (e.g., MemoryBank, A-MEM, Mem0, Zep) replay information drawn from dialogue histories and structured knowledge bases. Episodic memory maintains chronological records of specific experiences and their outcomes, while semantic memory stores abstracted, generalized knowledge such as facts, concepts, and rules. This hybrid approach—combining the ability to recall specific past events with access to general authoritative knowledge—mirrors the episodic–semantic distinction in human cognition and has proven highly effective for grounding agent reasoning. **Procedural memory** captures reusable action sequences and workflow knowledge, representing a third complementary dimension. Voyager's reusable code library exemplifies non-parametric procedural memory that accumulates skills without weight updates. Orthogonally, **short-term context optimization** methods (e.g., ACON, MEM1) manage information within a single task execution, while long-term memory systems persist knowledge across tasks.

Advanced memory architectures increasingly incorporate explicit retention and pruning policies. Systems like MemoryBank apply forgetting curves inspired by cognitive science, MemGPT implements hierarchical memory management analogous to operating system virtual memory (moving information between fast context and slow external storage), and MemInsight uses structured metadata to improve retrieval relevance. These mechanisms address a practical challenge that grows more acute in multi-agent settings: as the volume of stored experience scales with the number of agents and interactions, principled curation becomes essential to maintain retrieval quality. The empirical finding that memory failures increase with multi-agent scenario complexity—reaching 3.67 average failures in complex coordination tasks—underscores the urgency of this challenge.

Most existing memory systems target single-agent scenarios: Synapse uses successful past trajectories as exemplar prompts for computer control tasks, and Agent Workflow Memory (AWM) induces frequently used subtask sequences and consolidates them into reusable skills. Both demonstrate the value of procedural memory but do not address multi-agent settings, where memory must be distributed across agents with different roles and responsibilities.
