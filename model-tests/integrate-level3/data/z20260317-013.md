# Benchmarking and Evaluating Agentic AI Systems: Architecture, Safety, and the Case for Simplicity

Evaluating agentic AI systems requires moving beyond isolated component testing toward holistic assessment of how tool use, memory, coordination, and model selection interact under realistic conditions. The CLASSic framework provides a structured lens for this assessment, organizing evaluation across five dimensions—**Cost**, **Latency**, **Accuracy**, **Security**, and **Stability**—that together capture the full spectrum of deployment concerns. Two complementary evaluation perspectives illuminate these dimensions in practice: enterprise-focused benchmarks like AgentArch, which stress reliability and procedural compliance under production-like conditions, and broad academic evaluation suites that measure task performance and computational efficiency across diverse reasoning domains. Together, they expose a significant gap between the promise of agentic AI and its practical readiness—and, critically, they converge on a shared finding: single-agent or single-LLM execution frequently matches or outperforms multi-agent alternatives while achieving substantial cost reductions.

The challenges these benchmarks surface are not idiosyncratic to their designs; they reflect systemic failure patterns observed across the broader landscape of LLM-based agent development, where an analysis of 8,710 developer discussion threads has identified recurring categories of failures—logic, tool, performance, and compatibility—that map across the entire software development lifecycle.

### Agent Development as Software Development

The Software Development Lifecycle (SDLC) provides a systematic process for planning, designing, developing, testing, and maintaining software applications. In the context of LLM-based agents—increasingly regarded as a new form of software—agent development maps naturally onto traditional SDLC stages, though with distinctive characteristics at each phase. The process begins with **requirements analysis and design**, which defines the agent's functions, performance constraints, architecture, module composition, interaction flows, and data models. In practice, these two stages are often merged for agent development, as requirements directly shape system architecture decisions. The **implementation phase** transforms designs into functional agent systems through coding, model integration, and tool implementation. The **testing phase** ensures agents meet functional, performance, and security requirements, with special consideration for agent autonomy and potential unpredictability. The **deployment phase** delivers tested agent systems into production environments for stable, secure operation. The **maintenance phase** provides continuous post-deployment support including issue resolution, updates, and feature additions.

Mapping developer challenges to these SDLC stages reveals that different types of failures cluster at different phases. Logic-related failures (task termination, context misalignment, state inconsistency) primarily align with early design through deployment stages. Tool integration issues span implementation, testing, and deployment. Performance problems emerge from implementation through deployment. Version compatibility failures concentrate in deployment, testing, and maintenance stages. This temporal mapping enables both framework designers and developers to anticipate and address challenges specific to each development phase, and highlights that agent development introduces unique concerns—such as managing autonomous agent behavior, multi-agent coordination, and rapidly evolving dependency chains—that extend beyond traditional software engineering challenges. Enterprise benchmarks like AgentArch implicitly reflect this lifecycle perspective: their controlled evaluation isolates failures that arise primarily in the design-through-deployment stages (logic and coordination errors), while the developer-reported failures they corroborate span the full lifecycle including the maintenance-stage compatibility breakdowns that benchmarks cannot easily capture.

### Scope and Design of Enterprise Evaluation

AgentArch evaluates 18 distinct agentic configurations across six large language models—GPT-4.1, GPT-4o, GPT-4.1-mini, o3-mini, Claude Sonnet 4, and LLaMA 3.3 70B—on two enterprise workflows that differ in structural complexity. The simpler workflow (time-off requests) involves 8 tools and 3 agents; the more complex one (customer request routing) involves 31 tools and 9 agents. Each workflow includes 60 test samples with human-annotated ground truth covering expected tool inputs, outputs, and ordering.

A deliberate design choice distinguishes AgentArch from many existing benchmarks: the test data is constructed to be enterprise-realistic in its messiness. Knowledge base articles span thousands of words, tool responses return complex JSON objects where task-relevant information is buried within metadata and auxiliary fields, and the overall data is lengthy and noisy. This reflects production conditions where agents must extract signal from cluttered, verbose contexts rather than operating on clean, pre-structured inputs. The workflows also enforce highly specific directions requiring tasks to be completed in predefined order, reflecting enterprise constraints where even reaching the correct conclusion is insufficient—the system must select the right tools, invoke them with correct arguments, and reach the correct final outcome through the prescribed procedural path.

### Scope and Design of Multi-Domain Academic Evaluation

Complementing enterprise-focused evaluation, comprehensive academic benchmarking spans multiple task domains with carefully chosen metrics that capture both task performance and computational efficiency. A representative evaluation suite covers seven benchmarks across five domains: code generation (HumanEval with 131 test samples and MBPP with 341), mathematical reasoning (GSM8K with 1,055 samples and MATH with 486), general question answering (HotpotQA and DROP, each with 800 samples), domain-specific reasoning (Shopping-MMLU), and real-world planning with tool use (TravelPlanner with 180 samples).

Task-specific metrics include pass@1 accuracy for code generation, F1 score for question answering, solve rate for mathematics, accuracy for domain-specific multiple-choice tasks, and task success rate for planning. Computational cost is measured as USD token expenditure per workflow, accounting for both input and output tokens. Statistical reliability is ensured through three independent trials with reported means and standard deviations.

Shopping-MMLU introduces a domain-specific evaluation protocol using difficulty-based selection: from 33 shopping-related multiple-choice tasks, the 10 most challenging (where Claude 3.5 Sonnet achieved below 80% accuracy) are selected, covering product selection, sentiment analysis, attribute reasoning, multilingual query understanding, and other e-commerce reasoning domains. This difficulty-calibrated selection ensures the benchmark discriminates meaningfully between system capabilities rather than saturating on easy tasks.

Baseline methods in this academic suite span four categories: manual approaches (direct input-output prompting, chain-of-thought prompting, self-consistency, MultiPersona), automated multi-agent frameworks (AFlow, OneFlow), heterogeneous multi-agent systems, and single-LLM implementations of multi-agent workflows. The primary executor model is GPT-4o-mini (temperature 0), with robustness validation on Claude-3.5-Haiku and Qwen-3 8B (open-weight). Workflow optimization uses Claude-4.0-Sonnet as the designer with 20 optimization rounds.

### The Broader Accuracy Benchmark Landscape

Beyond AgentArch and the academic suite described above, accuracy evaluation for agentic systems has expanded into a rich ecosystem of specialized benchmarks that probe capabilities static question-answering cannot capture—success collapses when tasks require tool use, state tracking, and long-horizon recovery. **GAIA** benchmarks general assistants on tasks that are easy for humans but require multi-step decomposition and verification from AI. **OSWorld** and **OSWorld Verified** evaluate desktop control agents with reduced labeling noise, while **CoAct 1** combines computer-use actions with coding-as-action for verification and recovery. **SWE-bench** and **SWE-bench Verified** assess software engineering agents, with **SWE-bench Pro** pushing toward repository-scale longer-horizon tasks where context management becomes the central bottleneck. **τ-bench** evaluates multi-turn tool interaction under policies, **FrontierMath** targets hard-ceiling mathematical reasoning, and **AgentBench** evaluates across diverse environments. For multi-agent systems specifically, **MultiAgentBench (MARBLE)** measures emergent behaviors like negotiation efficiency and consensus formation. Modern evaluation increasingly reports compute budgets, run-to-run variance, and failure severity distributions alongside mean success—a trend that reflects growing recognition that accuracy alone is insufficient without understanding its cost, stability, and failure characteristics.

### Orchestration Strategies

AgentArch evaluates three orchestration strategies representing fundamentally different approaches to agent coordination. In the **orchestrator-led isolated agent** configuration, a central orchestrator assigns tasks to specialist agents and mediates all inter-agent communication—agents needing additional information must route requests through the orchestrator, creating a hub-and-spoke topology. In the **orchestrator-led open agent network**, the orchestrator assigns initial tasks but agents communicate directly with each other, selecting collaborators independently based on agent descriptions. This relaxes the communication bottleneck while retaining centralized task allocation. The **single agent** configuration gives one agent access to all available tools with no collaboration mechanisms, testing performance in larger, unfiltered option spaces where the agent must navigate the full tool inventory without the benefit of specialist decomposition.

These three strategies create a spectrum from maximal coordination overhead with narrow agent scope to minimal coordination with broad agent scope, enabling systematic investigation of how the division of labor interacts with model capability and task complexity. The academic evaluation suite provides a parallel comparison: automatically designed workflows executed by a single LLM are compared against multi-agent implementations of the same workflows, testing whether the coordination overhead of multiple agents provides genuine performance benefits beyond what a single model can achieve.

### Agent Styles: Function Calling vs. ReAct

Two agent execution styles are tested across all orchestration strategies. **Function calling** agents directly select tools using built-in API capabilities, while **ReAct** agents output explicit chain-of-thought reasoning before selecting actions in a structured thought-action-observation format. Function calling generally outperformed ReAct across most models, and the reasons appear to be structural rather than incidental: models proved more robust to hallucinations when using function calling schemas they were trained on compared to ReAct's JSON-based reasoning format.

A strong and practically important finding was that **multi-agent ReAct consistently underperformed across all models**. Hallucinations were found almost exclusively under ReAct settings, and the interaction between ReAct and multi-agent coordination proved particularly damaging. Sonnet 4 showed hallucination rates of 36% on ReAct in multi-agent systems but 0% in all other configurations. This pattern suggests that the combination of explicit reasoning traces and inter-agent communication creates compounding opportunities for fabrication—each agent's reasoning output becomes noisy context for downstream agents. This compounding effect connects to a broader pattern observed in developer practice: approximately 64% of redundant agent interactions in production systems stem from lacking message cooling mechanisms and relevance checks, causing agents to trigger duplicate responses or pass irrelevant context. The ReAct hallucination cascade is a specific, measurable instance of this general failure mode—where each reasoning trace becomes additional noisy context that downstream agents must process, amplifying rather than filtering signal.

The one notable exception was **LLaMA 3.3 70B**, which achieved its best score using single-agent ReAct—the only model to show this preference. This suggests that explicit reasoning may compensate for weaker native tool-use capabilities in some models, though LLaMA's overall performance remained poor (12.2% peak), limiting the practical significance of this finding. These results indicate that the relationship between explicit reasoning and task performance is complex and model-dependent, with the format and context of reasoning mattering as much as its presence.

The authors recommend avoiding multi-agent ReAct and suggest that future work should investigate whether applying ReAct selectively to only orchestration or only tool calling (rather than both) might recover some benefit of explicit reasoning without the reliability costs.

### Reasoning Scaffolding: Thinking Tools

Beyond the ReAct-vs-function-calling distinction, AgentArch investigates a subtler reasoning mechanism: **thinking tools**. These are pseudo-tools—a "math" tool for calculations and a "synthesize_collected_information" tool for data organization—that function as structured scratchpads. When invoked, whatever the model passes as the tool argument is simply returned as the result and added to memory, effectively giving the model space to generate extra reasoning tokens in the format of tool calls without actual tool execution. Examples of their use include calculating leave days across months (accounting for leap years) and synthesizing eligibility criteria from collected information.

Thinking tools proved beneficial for non-reasoning models on simpler tasks. GPT-4.1 improved dramatically from 48.5% to 70.8% with thinking tools enabled in single-agent function calling—the benchmark's highest overall score on the time-off workflow. GPT-4o rose from 26.9% to 39.0% in single-agent setups. However, thinking tools provided minimal benefit for o3-mini (a reasoning model), improving only from 55.8% to 56.7%, suggesting that models with built-in chain-of-thought reasoning capabilities gain little from external reasoning scaffolding.

Thinking tools also showed minimal impact on the complex customer routing task across all models, indicating their utility may be limited to tasks requiring specific calculated decisions—such as date counting and leave balance computation—rather than complex classification and routing decisions that depend more on contextual understanding and procedural navigation. This task-dependent pattern is important for deployment: thinking tools add latency (a cost the benchmark acknowledges but does not quantify), and their benefit appears confined to workflows with discrete computational steps rather than holistic judgment.

The contrast between thinking tools and ReAct illuminates different modes of reasoning augmentation. ReAct imposes a rigid thought-action-observation loop that restructures the entire agent execution flow, while thinking tools offer optional, on-demand reasoning within the existing function calling paradigm. The superior results from thinking tools suggest that giving models the choice of when to reason explicitly—rather than mandating it at every step—may be a more effective design pattern, at least for models that already possess competent function calling capabilities.

### Memory Management

Memory management was tested in two modes: **complete memory**, where agents see all previous tool calls, parameters, and responses, and **summarized memory**, where agents receive only final summaries from previous agents. Variation in memory management style had minimal impact on scores compared to other architectural dimensions such as orchestration strategy and agent style, though preferences varied by model and orchestration configuration. This finding is notable because it suggests that within stateless, single-execution contexts, the granularity of inter-agent information transfer matters less than the structural decisions about how agents are organized and how they reason.

However, the relative unimportance of memory granularity within single executions should not be extrapolated to longer interactions. Developer experience with production agent frameworks reveals that when conversations exceed 20 turns, some frameworks produce fragmented responses due to inadequate memory caching strategies, with state pointers being overwritten during concurrent access and temporary caches defaulting to local memory rather than persistent storage. AgentArch's test samples are single-execution workflows that do not probe these failure modes, meaning its memory findings speak to inter-agent information transfer within a bounded task rather than to the broader challenge of maintaining coherent state across extended interactions.

### Cost: The Efficiency–Intelligence Trade-Off

**Cost** reflects a fundamental efficiency–intelligence trade-off that pervades agentic system design. Hierarchical architectures like ReAcTree maximize reasoning depth but incur exponential increases in token consumption compared to linear chains or zero-shot prompting, creating a critical bottleneck for cost-sensitive real-time deployment. The KV cache mechanism provides a concrete, quantifiable lever for managing this trade-off, particularly in homogeneous single-LLM designs.

During autoregressive generation, the **key-value (KV) cache** stores previously computed key and value matrices from attention layers so that each new token generation does not require recomputing attention states for all preceding tokens—without caching, this recomputation would impose quadratic computational complexity. By retaining these intermediate states, KV caching enables significant speedups during sequential token generation.

The KV cache plays a central and underappreciated role in the analysis of single-agent versus multi-agent workflow execution, particularly for **homogeneous multi-agent workflows** where all agents share the same base LLM. In such workflows there is substantial contextual overlap between agent interactions: shared task descriptions, intermediate reasoning steps, and common knowledge. A single-LLM simulator can maintain one continuous conversation and reuse the KV cache across all agent turns, so that prefill cost scales only with the incremental growth of new tokens rather than the full accumulated prefix. Formally, if separate agent instances must each re-encode overlapping prefixes, the total cost is proportional to the sum of all prefix lengths across steps; with KV cache sharing, the cost reduces to the sum of only the new tokens added at each step, yielding savings whenever agent contexts overlap.

Experiments with both proprietary models (GPT-4o-mini, where KV cache costs were simulated based on final conversation states and official tokenization) and open-weight models (Qwen-3 8B with vLLM and explicit KV cache enabled) confirm these efficiency gains. With open-weight models, while multi-turn conversations increase average input token counts due to history accumulation, inference latency and throughput remain stable thanks to KV cache reuse. The cost reductions are substantial: single-LLM execution dramatically lowers token expenditure compared to multi-agent execution while maintaining comparable task performance.

This finding has direct implications for AgentArch's architectural comparisons. AgentArch's single-agent configuration—which already achieves the benchmark's highest scores in several settings—would benefit most from KV cache reuse, since one continuous conversation with a single model maximizes cache hit rates. Multi-agent configurations fragment the conversation across agent boundaries, and even when agents share the same base model, orchestration overhead (routing, summarization, inter-agent message formatting) introduces cache-breaking context switches. The token overhead AgentArch acknowledges but does not quantify in multi-agent designs is thus compounded by the loss of KV cache efficiency: not only do multi-agent systems require more inference calls, but each call is less efficient because it cannot fully leverage cached prefixes from prior turns handled by different agent instances.
