# Memory Poisoning, RAG Security, and Defenses for LLM Agents

Large language model agents with persistent memory are vulnerable to **memory poisoning**—adversarial injection of malicious instructions into long-term memory through ordinary interactions. The most studied technique, **MINJA**, uses **bridging steps** (gradually steering toward a poisoned association), **indication prompts** (signaling the interaction should be memorized), and **progressive shortening** (compressing malicious context into plausible entries). Under empty-memory conditions, MINJA achieves >95% injection success rate (ISR) and >70% attack success rate (ASR).

### RAG and Memory-Augmented Systems as Attack Surfaces

Retrieval-augmented generation systems face distinctive corpus-integrity threats. **BadRAG** demonstrates that crafted malicious documents can override legitimate retrieved content. **PoisonedRAG** shows corpus injection corrupts available knowledge. **AgentPoison** targets both RAG knowledge bases and agent memory, assuming stronger attacker capabilities including direct access. Related threats include FuncPoison, contagious jailbreak in multi-agent systems, and sleeper agent/instruction-tuning backdoors.

In memory-augmented agents, the retrieval mechanism critically determines both attack success and defense. Levenshtein distance-based retrieval surfaces memories matching query identifiers—when sufficient legitimate memories exist for an entity, poisoned entries containing extraneous identifiers score lower and are less likely retrieved. The **retrieval window size** (top-N) directly affects vulnerability: larger windows increase the probability poisoned entries surface alongside legitimate ones.

### EHR Systems: A High-Stakes Case

Electronic Health Record agents—with shared persistent memory across clinicians—represent a particularly dangerous context. The critical attack is **patient identifier swapping**: injecting instructions mapping one patient's ID to another's records (e.g., victim 027-22704 → target 015-91239 in MIMIC-III), risking misdiagnosis. Indication prompts mimic plausible administrative operations (record migrations, ID consolidations), evading simple content analysis.

### Degradation Under Realistic Conditions

Pre-existing legitimate memories substantially degrade attacks. Six correct memories reduce ISR from 100% to ~25% and ASR from ~62% to 6% (GPT-4o-mini) or 0% (Llama-3.1-8B-Instruct), as similarity-based retrieval preferentially surfaces legitimate entries. Expanding retrieval windows from 3 to 10 restores ISR to 100% for GPT-4o-mini and raises Llama ASR to ~27%. Production systems with millions of records make achieving sufficient poisoned-to-legitimate ratios extremely difficult; best-case realistic ASR is ~38% (GPT-4o-mini) and ~28% (Llama), and the spamming needed to shift ratios is itself detectable.

### Defense Mechanisms

**Input/output moderation** applies two-stage gating: input-stage static heuristics and keyword matching (detecting "Knowledge:", "refer", control characters, EHR patterns like "Refer X to Y") escalate to semantic LLM classification; output-stage evaluation checks for dangerous code operations and relevance. Multiple orthogonal signals produce a composite trust score; memory appends require exceeding a conservative threshold. Related approaches include DRIFT (rule-based input filtering, not addressing memory-level threats) and A-MemGuard (access control requiring direct memory instrumentation unavailable in black-box settings).

**Trust-aware retrieval and memory sanitization** operate at append and retrieval time. Append-time trust scores incorporate code safety, semantic relevance, answer verification, optional sandbox re-execution, and poisoning template detection. Retrieval-time temporal decay down-weights older entries; those below threshold are excluded, with remaining entries ranked by combined similarity and trust.

### The Trust Score Failure Mode

Trust-based defenses expose a fundamental tension. GPT-4o-mini rejects all candidate entries (mean trust ~0.28)—uncontaminated but empty memory sacrificing utility entirely. Gemini-2.0-Flash shows polarized trust: 82 entries accepted at 1.0, 69 rejected at ~0.35—but 54 of the 82 accepted were confirmed malicious, having manipulated the agent's own trust assessment. Once poisoned entries receive maximum trust, retrieval-time filtering fails. Trust-based defenses function as **confidence filters, not security filters**: the same in-context learning capacity making a model capable also makes it exploitable through its own confidence heuristics.

### Design Implications

**Memory density** is a first-order security parameter. Aggressively populating memory with verified execution traces during normal operation builds resilience inadvertently—procedural memory accumulation simultaneously improves performance and hardens against injection.

**Retrieval window size** presents a genuine security–utility tension unresolvable by retrieval design alone, requiring co-consideration of memory curation, write-time anomaly detection, and provenance tracking.

**Model-dependent vulnerability**: GPT-4o-mini vs. Llama differences indicate poisoning susceptibility depends on how strongly in-context learning weights retrieved examples. Models that more rigidly follow few-shot patterns are simultaneously more capable and more vulnerable—a fundamental adaptability–robustness tension.

**Decoupling trust from safety** through external verification—validating patient ID redirects against immutable ground-truth databases—is the most promising direction for catching high-confidence poisoning that fools the model's own assessment. Any defense relying on the attacked model to judge attack legitimacy inherits the model's blindspots.
