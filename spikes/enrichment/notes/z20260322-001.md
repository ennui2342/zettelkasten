# Multi-Agent Architectures: Topology, Organization, and Workflow Automation

Multi-agent systems (MAS) built on LLMs automate complex workflows by organizing specialized agents—each with distinct prompts, tools, and roles—into structured interaction topologies. As tasks exceed a single model's capacity, distributing work across collaborating agents becomes essential. These systems fall into two primary architectural families—**centralized** and **decentralized**—though effectiveness depends not merely on which family is chosen but on fine-grained implementation decisions within that family: interaction topology, foundational paradigms (role-playing, tool-augmented collaboration, multi-agent debate), the degree of automation in system design, and several key design primitives governing agent autonomy, planning strategy, and context organization.

### Architectural Families: Centralized and Decentralized

In **centralized architectures**, a single coordinating agent (orchestrator) decomposes a user's request into a plan, assigns subtasks to specialized sub-agents, and aggregates outputs. Frameworks such as Magentic-One, LangGraph, and OpenAI Agents provide abstractions for building centralized MAS, though they differ substantially in implementation details. In **decentralized architectures**, decision-making authority is distributed across agents via peer-to-peer communication and local coordination, reducing reliance on any single agent but raising risks of mis-coordination or conflicting actions.

Empirical comparisons reveal that centralized MAS are not inherently safer than single-agent setups—strong planning and execution capabilities can efficiently realize harmful goals when alignment fails. Decentralized MAS tend to perform better on context-heavy tasks because individual agents retain full context and can independently reject unsafe actions, though this advantage is domain-dependent. Critically, implementation choices within the same architectural family (e.g., two centralized systems with different planning strategies or autonomy levels) produce dramatically different security and performance outcomes, demonstrating that high-level architectural category alone is insufficient for predicting system behavior.

### Topology Design and Trade-offs

Within these broad families, three canonical topologies capture the dominant structural trade-offs:

**Star (hub-and-spoke).** A central aggregator receives from all leaf agents in a single hop, minimizing communication depth. By the data processing inequality, this is information-dominant when unconstrained—hierarchical aggregation can only discard information. However, the context window constraint $n \cdot m \leq W$ caps incorporated agents at $\lfloor W/m \rfloor$; beyond this, additional budget can only strengthen existing agents or increase message length, not add parallel evidence.

**Chain (sequential).** Each agent's output feeds the next, avoiding fan-in bottlenecks but suffering compounding communication loss: binary bias decays as $\eta^L$, continuous MSE grows linearly with length. Chains are beneficial only when intermediate nodes perform substantive new computation (tool use, retrieval, verification) rather than relaying information.

**Hierarchical tree.** Distributes aggregation across small nodes each respecting local fan-in $km \leq W$, enabling $n = k^d$ leaves through depth $d$—the minimal structure scaling beyond the star's context cap.

**Mesh (swarm/decentralized).** Agents interact dynamically without central control, optimizing for creativity and diversity. Multi-agent debate improves convergence on correct answers through mutual critique; adaptive variants add adversarial roles (Devil's Advocate). Economic and social simulations (TradingAgents, Generative Agents, SocioVerse) use mesh topologies to model emergent phenomena like price discovery and norm propagation across thousands of agents.

### Design Primitives Shaping MAS Behavior

Several key design primitives shape MAS behavior and vulnerability profiles, cutting across topological choices:

- **Sub-agent autonomy** refers to the granularity of task delegation. Orchestrators may assign high-level subtasks (e.g., "post this content") requiring multiple actions, or atomic instructions (e.g., "click this button") maintaining fine-grained control. Higher autonomy preserves context for sub-agents and enables richer local reasoning, while atomic delegation fragments intent across isolated actions and keeps tighter orchestrator control.
- **Stratified vs. combined planning** determines whether plan construction occurs as a distinct phase before execution (stratified) or is integrated with delegation decisions (combined). Stratified planning enables focused task decomposition but requires additional logic for handling failures and replanning; combined planning allows dynamic adaptation but increases cognitive load on the orchestrator.
- **Context organization** governs how task descriptions, plans, and agent roles are structured within the LLM's input context. Different frameworks sequence plan information, task descriptions, and role specifications differently (e.g., in system messages, user messages, or assistant messages), influencing how the underlying LLM processes and prioritizes information. This interacts directly with topology: star topologies concentrate context at the hub, while mesh topologies distribute it.

### From Open-Ended Chat to Workflow Graphs

Production systems have shifted from open-ended multi-agent chat loops toward explicit workflow graphs modeled as state machines—nodes represent tool calls or LLM invocations, edges represent permissible transitions. These are formally **directed conditional graphs** with state transition functions capturing cyclical refinement, conditional branching, and fault-tolerance. LangGraph operationalizes this via graph traversal with state persistence, checkpoints, and controlled cycles. OpenAI Swarm provides a lightweight complement emphasizing agent handoffs. Graph boundaries define possible actions, reducing runaway loops and interacting naturally with safety evaluation.

The orchestration loop follows a turn-based protocol: a scheduler selects the active agent, task agents execute via tool-use commands against sandboxed environments, observations flow back, and re-planning triggers when progress stalls. Whether this loop employs stratified or combined planning significantly affects both robustness and the system's vulnerability surface.

### Hierarchical Organization

The dominant centralized pattern uses a supervisor/orchestrator decomposing goals into subtasks delegated to specialized workers, typically in two layers: strategic supervision and tactical execution. MetaGPT encodes Standard Operating Procedures into agent prompts with roles (Product Manager, Architect, Engineer) producing standardized deliverables in a waterfall sequence. ChatDev simulates an entire software company with self-organizing phases. Variations include feedback-driven replanning (ProSEA), predefined centralized workflows (AutoAct), and DAG-based "agent-as-a-tool" hierarchies (InfiAgent). Hierarchical multi-agent reinforcement learning confirms sub-task-level coordination is more efficient than flat primitive-action coordination.

For error mitigation, cross-examination patterns where Verifier agents challenge Worker outputs enable million-step reasoning with near-zero error accumulation. Supervisory architectures using Reviewer agents to filter outputs dramatically reduce hallucination. These verification mechanisms interact with the autonomy primitive: high-autonomy sub-agents benefit more from independent verification, while atomic-delegation systems rely on the orchestrator's own coherence checks.

### System Automation Spectrum

Systems range across: **Handcrafted** (AutoGen, ChatDev)—fully manual; **Partially automated**—automating roles, prompts (DSPy, TextGrad), or topologies (GPTSwarm, G-Designer); **Fully automated**—autonomous design of all modules. Prompt optimization is often more impactful than structural changes. Dynamic contribution estimation can mute unhelpful agents to reduce token usage.

### Homogeneous vs. Heterogeneous Workflows

Most MAS are **homogeneous** (same base LLM, differing in prompts/tools/position). Under mild conditions—deterministic tool effects, history-dependent routing, deterministic decoding—a single-LLM simulator induces identical transcript distributions. Empirical validation across coding, math, QA, and planning benchmarks confirms single-agent execution matches multi-agent performance. The key mechanism is **KV cache reuse**: one continuous conversation state avoids redundant re-encoding of overlapping prefixes, yielding dramatic inference cost reductions.

**Heterogeneous** workflows leverage diverse LLM capabilities but cannot share KV cache states across models—a fundamental coordination cost. Performance tends to be bounded by the best constituent homogeneous workflow, though heterogeneous systems enable cost-performance optimization through intelligent routing.

### Complexity Matching and Memory

Complex multi-agent frameworks on simple tasks can be counterproductive—architectural overfitting introduces harmful overhead. Current systems are largely stateless. Integrating **procedural memory** allows agents to accumulate execution knowledge; smaller teams with memory can match larger LLM-only teams without it. Architectural simplicity combined with memory augmentation and heterogeneous model selection can outperform brute-force scaling. Benchmarks like OfficeBench provide standardized evaluation with programmatic state verification.

The broader lesson is that neither architectural family nor topology alone determines system quality. The interplay of autonomy granularity, planning strategy, context organization, memory, and model composition creates a high-dimensional design space where seemingly minor implementation choices produce dramatically different outcomes in both capability and safety.
