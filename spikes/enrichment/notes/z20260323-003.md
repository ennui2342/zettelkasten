# Empirical Software Engineering Research Methods

Empirical investigation of software development practices involves systematic collection and analysis of real-world developer data to understand how tools, frameworks, and processes are used in practice. Large-scale studies of open-source repositories and developer discussions provide evidence-based insights that complement theoretical analysis.

A multi-stage data collection methodology can be employed to study framework ecosystems. Repository selection criteria typically include keyword filtering strategies (compound keywords to balance coverage and relevance), documentation completeness requirements, community attention thresholds (minimum stars and forks), and recency constraints. Data cleaning processes involve removing duplicates (including cross-posted discussions), filtering invalid content (empty messages, emoji-only posts, purely promotional comments), and excluding bug reports lacking sufficient context.

For analyzing unstructured developer discussions at scale, a processing pipeline combining automated and manual methods proves effective. Term frequency-inverse document frequency (TF-IDF) algorithms extract representative keywords from discussion threads. Large language models such as GPT-4o can perform semantic classification, mapping threads to standardized functional tags and unifying different expressions into a consistent semantic space. Automated inductive summarization transforms multi-turn, lengthy discussion threads into concise representations of developer intent, using carefully designed system and user prompts with controlled generation parameters (low temperature, top-1 sampling) to ensure reproducibility.

Open coding procedures enable qualitative analysis, where textual excerpts are labeled with descriptive codes capturing core meaning. Inter-rater reliability is quantified using Cohen's kappa, with substantial agreement thresholds ensuring classification quality. Framework identification combines topic tag frequency analysis with manual relevance verification, and dependency analysis examines requirements.txt and package.json files to verify whether topic tags accurately reflect actual framework usage.

Threats to validity in such studies include potential model-generated bias in automated classification, the limitation of studying only open-source projects (excluding closed-source enterprise environments where framework preferences may differ), and the possibility that topic tags and dependency files may not fully reflect actual usage, particularly for implicit dependencies or customized packages. Multi-source validation combining framework documentation, example code, and developer discussions helps mitigate these limitations.
